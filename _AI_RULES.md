# Технические требования к React-проекту (AI-Optimized)

1. Архитектура проекта и файловая структура

1.1. Структура директорий:

src/components/ui/ — Базовые атомарные компоненты (Button, Input, Icon). Без бизнес-логики.

src/components/blocks/ — Крупные блоки страницы (Hero, Features, ContactForm). Собираются из ui/.

src/layouts/ — Обертки страниц (MainLayout, AuthLayout).

src/data/schemas/ — JSON-схемы для валидации контента страниц.

src/types/ — Глобальные TypeScript интерфейсы.

1.2. Принцип именования:

Папки компонентов: PascalCase (например, src/components/blocks/HeroSection/).

Файл компонента: index.tsx внутри папки компонента.

Именование пропсов: camelCase.

Именование обработчиков событий: префикс on (например, onSubmit, onClick).

1.3. Экспорт:

Использовать Named Exports (export const HeroSection = ...), избегать export default.

Все блоки должны быть переэкспортированы в едином файле src/components/blocks/index.ts для упрощения импорта динамическим рендерером.

2. Стек и Типизация (TypeScript)

2.1. Строгий режим:

В tsconfig.json установить "strict": true.

Запрещено использование типа any.

2.2. Интерфейсы компонентов (Props):

Каждый компонент должен иметь экспортируемый интерфейс пропсов (например, export interface HeroSectionProps).

Интерфейс должен быть объявлен перед функцией компонента в том же файле.

Обязательно использование JSDoc комментариев для каждого пропса, описывающего контент (например, /** Заголовок H1 */ title: string;).

2.3. Union Types (Ограничение значений):

Для стилистических вариаций (цвет, размер, выравнивание) использовать строгие объединения строк, а не string.

Неверно: variant: string

Верно: variant: 'primary' | 'secondary' | 'outline'

3. Компоненты и Рендеринг (Page Builder Pattern)

3.1. Изоляция данных:

Компоненты в src/components/blocks/ должны быть "глупыми" (Presentational Components).

Запрещены прямые API-запросы внутри блоков. Данные передаются только через props.

3.2. Динамический рендеринг:

Страницы должны собираться через единый компонент <PageBuilder blocks={data} />.

Структура данных страницы (JSON) должна соответствовать типу:

```ts
type BlockData = {
  id: string;
  type: keyof typeof BlockComponents; // Строгое соответствие имени компонента
  props: Record<string, unknown>; // Пропсы конкретного блока
};
```

3.3. Условный рендеринг:

Использовать тернарные операторы condition ? <A/> : <B/> или condition && <A/>.

Избегать сложной логики (if/else) внутри JSX.

4. Стилизация (Tailwind CSS)

4.1. Ограничение произвольных значений:

Запрещено использование arbitrary values (например, w-[350px], bg-[#123456]).

Использовать только классы из дизайн-системы, прописанные в tailwind.config.js.

4.2. Отступы и Layout:

Корневой элемент блока не должен иметь внешних отступов (margin).

Отступы между блоками контролируются родительским контейнером (например, через space-y-* или gap-*).

Блоки должны занимать 100% ширины родителя (w-full).

4.3. Динамические классы:

Использовать библиотеку clsx или classnames для условного применения классов.

Запрещена конкатенация строк для классов (например, className={"btn " + variant}).

5. Управление контентом (Content as Code)

5.1. Формат данных:

Контент страниц кейсов должен храниться в отдельных JSON или TS файлах в src/content/.

Изображения должны передаваться как объект: { src: string, alt: string, width?: number, height?: number }.

5.2. Rich Text:

Для текстовых полей с форматированием использовать только ограниченный набор HTML-тегов или Markdown, передаваемый как проп (например, descriptionMarkdown). Запрещен dangerouslySetInnerHTML без предварительной санации.

6. Примеры для AI (Few-Shot Prompting)

6.1. Файл-образец:

В корне проекта создать файл _AI_RULES.md или _EXAMPLE_BLOCK.tsx, содержащий эталонный код компонента-блока.

Файл должен содержать пример:

Импортов.

Интерфейса с JSDoc.

Компонента с деструктуризацией пропсов.

Использования Tailwind классов.

Конец документа требований.
